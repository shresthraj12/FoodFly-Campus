FoodFly Campus

Smart Campus Food Delivery Platform

A high-availability, multi-container student delivery platform engineered for LPU hostels. Built with Node.js, React, and MongoDB, orchestrated with Docker, automated via Jenkins pipelines, and monitored with Prometheus & Grafana.

Deployments: Docker Compose
Observability: Prometheus & Grafana
02

The Problem Statement

📍

Hostel Gate Restrictions

Delivery agents from external platforms (Zomato/Swiggy) are stopped at the main gates of LPU. Students must walk up to 1.5 km just to receive their food packages.

⏱️

Long Delivery & Cold Food

Due to the massive student population and campus checkposts, standard riders face route confusion, resulting in extreme delays and cold, unappetizing meals.

🛒

Fragmented Campus Canteens

LPU campus contains multiple individual tuck shops, canteens, and food courts (like block canteens). There is no unified system to order items from multiple shops in a single checkout.

03

The Solution Overview

Platform Highlights

Hyper-Local Campus Platform

FoodFly Campus resolves campus restrictions by creating a dedicated, closed-loop ordering network within the university premises.

🏫

Block-Wise Delivery Drop-Offs

Riders deliver straight to designated tables at BH-1, BH-2, GH-1, etc.

⚡

Student & Internal Riders

Optimized internal routing paths using campus delivery agents with full permission to access hostel lobbies.

⚙️

DevOps Infrastructure

Orchestrated with Docker Compose and managed with continuous automation to handle heavy peak order rushes.

06

System Architecture

Unified ingress routes clients via Nginx Reverse Proxy (port 8080) to either the static React content or Express REST API. Jenkins automation pulls GitHub changes, tests, compiles, and deploys using Docker Compose. cAdvisor and Node Exporter collect metrics scraped by Prometheus and graphed in Grafana dashboards.

07

Docker & Containerization

docker-compose.yml 
services:
  mongodb:
    image: mongo:latest
    container_name: foodfly_mongodb
    volumes: [mongodb_data:/data/db]
  
  backend:
    build: ./backend
    container_name: foodfly_backend
    depends_on: { mongodb: { condition: healthy } }
    environment: [MONGODB_URI, JWT_SECRET]
  
  frontend:
    build: ./frontend
    container_name: foodfly_frontend
    ports: ["3000:80"]

  nginx:
    image: nginx:alpine
    ports: ["8080:80"]
    volumes: [./nginx/nginx.conf:/etc/nginx/nginx.conf]

🐋 Bridge Network Isolation

All microservices are isolated inside a custom Docker bridge network. Services connect securely using container hostnames (e.g. mongodb://mongodb:27017), hiding ports from the public internet.
